## Block 3 — Work-first soupis verification

**Status:** verified (read-only) on `claude/intelligent-keller-GGWBt`.

**Verdict: PARTIAL** — the *work-first data model* for `create_work_breakdown`
exists, is wired, and is covered by a test; but *catalog-last staging* (a
separate frozen step per Pattern 15 / the W2 StageGating MVP) is **not**
implemented in this tool. Catalog matching is interleaved in the same call.

### Evidence (symbol + test)

**Tool:** `create_work_breakdown` —
`concrete-agent/packages/core-backend/app/mcp/tools/breakdown.py:47`
(async fn).

What is DONE (work-first capability):
- Work items are decomposed from element templates in **Step 3**
  (`breakdown.py:133-159`) independently of any catalog — quantity is derived
  from geometry (`volume`, `formwork_area`, `rebar_tons`, …), not from a code.
- Catalog code + price are **optional**, added only when a match is found:
  `if otskp_match:` → `item["otskp_code"]`, `unit_price_czk`, `total_price_czk`
  (`breakdown.py:181-186`). The base item always carries only
  `work_description / unit / quantity / hsv_section / element_name /
  element_type` (`breakdown.py:173-180`).
- Catalog lookup is itself gated: `if catalog in ("otskp", "both")`
  (`breakdown.py:163`). Any other `catalog` value yields a **code-less,
  price-less work list** — i.e. a usable work-first output as a side effect.
- **Wired:** registered `mcp.tool()(create_work_breakdown)`
  (`app/mcp/server.py:87`); REST wrapper `routes.py:1399-1400`; billed
  20 credits `auth.py:59`.
- **Tested:** `test_breakdown_basic`
  (`concrete-agent/packages/core-backend/tests/test_mcp_compatibility.py:518`).
  Asserts `items / total_items > 0` and that each item has
  `work_description, unit, quantity, element_type` — it does **not** require
  `otskp_code`, which confirms the code-less contract is intentional.

What is MISSING (catalog-last staging):
- Matching runs **inside the same call**, interleaved per work item
  (`breakdown.py:161-186`) — there is no separate "freeze the list → STOP
  gate → then match" stage and no frozen-snapshot artifact between work
  assembly and catalog matching, which is the core of Pattern 15 / W2 AC#19.
- There is no explicit work-only mode (e.g. `catalog="none"`); suppressing
  codes today relies on passing a `catalog` value outside `{otskp, both}`,
  which is undocumented behavior rather than a first-class stage.

**Conclusion for W2 StageGating MVP:** the data model already permits
code-less work items (so no schema change is needed), but the *staged*
work-first → catalog-last sequence with an explicit gate must still be
built; `create_work_breakdown` currently does both in one pass.
