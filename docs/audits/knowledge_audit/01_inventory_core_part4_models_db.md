## CORE-wide observations

1. **Knowledge base is well-organized but uneven**
   - B1, B2, B3, B4, B9 are fleshed out and actively used (40+ JSON files between them)
   - B5–B8 are placeholders (4 empty stubs)
   - Root `all_pdf_knowledge.json` (12,837 lines) is the master extraction dump generated by `extract_all_pdfs.py` + MinerU; not manually edited

2. **Prompts split into two parallel philosophies**
   - `prompts/roles/*.md` — long-form persona prompts (1,000–1,500 lines each), used by multi-role audit pipeline
   - `prompts/claude/*` and `prompts/gpt4/*` — task-specific prompts (one parser, one OCR, etc.)
   - Two of the role prompts (`structural_engineer.md`, `standards_checker.md`) **embed** large chunks from `B2_csn_standards/VL_4_2021_Mosty_markdown.md` and `tkp_18.md` — couples prompts to KB

3. **Hardcoded constants need extraction (8 hotspots — see part 3)**
   - All 8 are confidence thresholds or critical-field lists
   - Recommended target: existing `app/core/config.py` (Python module already loaded as runtime config)

4. **Duplicates to resolve inside CORE itself**
   - `prompts/master_framework.txt` ↔ `prompts/resource_calculation/master_framework.txt`
   - `prompts/claude/assistant/construction_expert.txt` ↔ `prompts/claude/assistant/stav_expert_v2.txt`
   - `B3_current_prices/all_pdf_extractions.json` ⊂ `knowledge_base/all_pdf_knowledge.json` (subset relationship)

5. **MCP tool surface = 9 tools backed by these modules**
   The 9 MCP tools (`find_otskp_code`, `find_urs_code`, `classify_construction_element`, `calculate_concrete_works`, `parse_construction_budget`, `analyze_construction_document`, `create_work_breakdown`, `get_construction_advisor`, `search_czech_construction_norms`) wrap services covered above. After consolidation, MCP wrapper signatures must remain stable — see root `CLAUDE.md` "MCP Compatibility Check" rules.

6. **Cross-zone dup hints from CORE perspective**
   Listed inline in parts 1–3; full triangulation in `06_duplicates_conflicts.md`. Quick summary:
   - OTSKP catalog → 3 places (CORE XML, URS_MATCHER SQLite + CSV)
   - URS catalog → 2 places (CORE B1_urs_codes + URS_MATCHER URS201801.csv)
   - Concrete classes / exposure → 4 places (CORE csn_en_206.json, Monolit maturity.ts, Monolit concreteExtractor.js, URS_MATCHER norms/knowledgeBase.js)
   - Curing days → 2 places (CORE tkp_18.md, Monolit maturity.ts CURING_DAYS_TABLE)
   - Formwork systems → 4 places (CORE B3 doka+peri JSONs, Monolit formwork-systems.ts, Registry formwork_knowledge.json)
   - Pump specs → 3 places (CORE B9 pumps.json, Monolit pump-engine.ts, Registry pump_knowledge.json + pump_suppliers.json)
   - Productivity / labor rates → 3 places (CORE B4, Monolit constants, URS_MATCHER norms)
   - 11 BOQ skupiny / classification rules → 3 places (CORE work_classifier YAML, Registry frontend rules.ts, Registry backend rules.ts)
