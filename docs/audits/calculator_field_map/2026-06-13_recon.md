## 2. Разрывы связности (главная боль) — диагноз Alexandra подтверждён

**(a) Где считается V/S ОДНОГО элемента.** Два узких фрагментарных пути, оба `useCalculator.ts:658–698`:
- горизонтальные блоки фундаментов (~7 типов, gate `CalculatorFormFields.tsx:194–198`): `L×W×H → volume_m3`, `2(L+W)×H → formwork_area_m2`; только при `volume_mode='from_geometry'`;
- pilota: `n·π(Ø/2)²·L → volume_m3`.
Все **остальные ~16 типов** не имеют пути размеры→объём — `volume_m3` и `formwork_area_m2` вводятся руками.

**(b) Почему НЕТ агрегации нескольких элементов — структурный блок.** `planElement(input: PlannerInput)` (`planner-orchestrator.ts:841`) принимает **РОВНО ОДИН элемент за вызов.** В `PlannerInput` (`:60–379`) **нет массива элементов.** «Множественность» только: `num_identical_elements` (копии того же), `num_bridges` (тот же мост ×N), `num_dilatation_sections`/`manual_zabery` (подразбивка того же залива). Фронт зеркалит 1:1: `form` = один `FormState`, `buildInput` = один `PlannerInput`, `runCalculation` → `planElement(input)` один раз (`:991`). **Списка элементов в форме нет, кнопки «добавить элемент» нет, суммы по проекту нет** — проектные итоги живут в другой поверхности (positions-таблица/`FlatKPIPanel`, вне этих файлов).

**(c) Поля разбивки на такты — изолированы от геометрии.** Блок «Členění konstrukce» (`CalculatorFormFields.tsx:648–992`) пишет `has_dilatation_joints`/`num_dilatation_sections`/`dilatation_spacing_m`/`tacts_per_section_*`/`manual_zabery`. Ни одно **не читает** L×W×H. Единственная слабая связь: `dilatation_spacing_m` derive из `total_length_m` — но `total_length_m` редактируется **только внутри самого блока дилатаций** (`:741–749`), это **другое число**, чем `length_m_input` геометрии. Итог: «N элементов с тактами» **невыразимо** — либо один элемент с тактами, либо N идентичных копий (без per-copy тактов), и никогда N разных элементов с суммой.

## 3. Как устроен вход движка СЕЙЧАС — факт (гипотеза Alexandra подтверждена)

- `export function planElement(input: PlannerInput): PlannerOutput` — `planner-orchestrator.ts:841`. **Один вход → один выход.**
- `PlannerInput` (`:60–379`) — плоская одно-элементная форма: `volume_m3: number` (`:69`, единственная обязательная величина кроме `has_dilatacni_spary:boolean` `:109`); идентификация `element_name?` ИЛИ `element_type?` (`:62–65`); throw если нет ни одного (`:888`). **Нет `elements: PlannerInput[]`, нет массива, нет project-обёртки.**
- Backend `/api/calculate` (`engine.js:47–82`) — тонкий pass-through: `res.json(planElement(input))` (`:76`), один элемент. Batch-эндпоинта нет.

**Вывод: один элемент за вызов — это структурный корень проблемы агрегации, подтверждён типом и сигнатурой, не предположение.** Лечение агрегации = «научить вход принимать список / ввести project-обёртку + агрегатор», а НЕ «добавить поле суммы».

## 4. MCP vs фронт parity (разошлись)

Оба POST'ят в один `/api/calculate`, но **поверхности параметров разные**:

**В MCP, но нет как поля фронта:** `width_m` (hint→nk_width_m/area, `calculator.py:252–264`); `formwork_length_bm`+`cycle_length_bm` (bm-количество + říms-záběr, `:266–276`) — у фронта **нет bm-длины/cycle вообще**; `tz_technology/tz_pour_stages/tz_quote/tz_anchor` (`:321–324`) — фронт строит `tz_facts` иначе (из текста TZ через `extractConstructionTechnology`, `useCalculator.ts:1292–1305`); `nk_subtype` (строковый словарь) vs фронтовый `bridge_deck_subtype` (enum); `_ENGINE_UNSUPPORTED_TYPES` refusal (`:45–47`) — MCP-only.

**Во фронте, но MCP не выставляет:** все ресурсы (crews/wages/sets/shift), все tact/section-контролы (`num_dilatation_sections`, `tacts_per_section`, `tact_volumes`, `adjacent_sections`, `scheduling_mode_override`, `working_joints_allowed`), `rebar_mass_kg`/`rebar_diameter_mm`, `total_length_m`, `deck_thickness_m`, `lost_formwork_area_m2`, `target_pour_window_h`, `concrete_consistency`, `exposure_classes[]` (MCP шлёт только singular), `enable_monte_carlo`, `deadline_days`, `formwork_shape_correction`, `num_identical_elements`/`formwork_sets_count`, `mss_tact_days`, большинство pile-полей (length/casing/rebar_index/rates/cap).

Итог: MCP — узкая hint-поверхность (volume + класс + пара гео-hint'ов + tz_facts); фронт — полная ресурс/scheduling-поверхность. **Не в parity.** Оба — одно-элементные.

## Вывод — топ структурных проблем по приоритету (ТОЛЬКО диагноз)

1. **`planElement` одно-элементный по типу** (`planner-orchestrator.ts:841`; в `PlannerInput` нет массива). Корень «нельзя описать проект / сложить общий бетон». Всё остальное — следствие. **Высший приоритет.**
2. **Геометрия→количество фрагментарна и изолирована от тактов.** L×W×H только ~7 типов; `length_m_input` геометрии не кормит `total_length_m` тактового движка → размеры и разбивка реально разъединены (точная жалоба Alexandra). **Второй.**
3. **Множественность выражена тремя несовместимыми механизмами** (`num_identical_elements` ⊥ `num_dilatation_sections` mutex; `manual_zabery` схлопывается в count+max). «N разных элементов, каждый со своими тактами» невыразимо. **Третий.**
4. **Legacy tact-поля полу-подключены и активно вводят в заблуждение** (advisor читает, WizardHints пишет, `buildInput` игнорирует → тихая потеря данных). **Четвёртый.**
5. **Мёртвые/неотправляемые поля подрывают доверие** (`price_crane_czk_shift`, `price_pump_czk_h` собираются и не шлются; `include_kridla`/`kridla_height_m` display-only) — буквально «поля, добавленные по ходу без эффекта». **Низший структурный, но самый видимый пользователю.**
