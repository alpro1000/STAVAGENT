# TASK — Тонкий гибрид-оркестратор конвейера (Вариант B из ADK-спайка)

**Решение ADK-спайка:** не брать ADK, построить свой тонкий гибрид и переиспользовать
существующее. Этот документ — само задание.

**Суть:** заменить заглушку диспетчеризации в `stage_gating` (шаг WORK_ATOMIZATION)
реальным детерминированным рецептом по MCP-тулзам, с **одним LLM-вызовом-решением**
в точках нюанса, прокидывая контекст объекта. Переиспользуем `stage_gating` (сессии/
аудит/HITL/replay), существующий разумник и MCP-тулзы — не строим заново.

---

## 1. Коллизия имён (важно)
В репо УЖЕ есть `orchestrator_hybrid.py` — это Multi-Role разумник (Vertex Gemini).
Новый оркестратор — это **другое**: диспетчер конвейера WORK_ATOMIZATION. Назвать его
отлично от `orchestrator_hybrid`, чтобы не путать. Сам `orchestrator_hybrid` можно
**переиспользовать** как механизм LLM-решения (см. §3).

## 2. Что переиспользуем (НЕ переписываем)
- `stage_gating`: durable-сессии, append-only аудит, HITL, replay — это актив. Только
  **заполнить заглушку диспетчеризации**, сессии/аудит не менять.
- MCP-тулзы: `detect_object_type`, `classify_construction_element`,
  `create_work_breakdown`, `calculate_concrete_works`, `export_soupis`.
- Существующий разумник (`orchestrator_hybrid`, Vertex Gemini) — для LLM-решения в нюансе.
- W3b `detect_object_type` + кэш `object_types` — для контекста объекта.

## 3. Механизм
Детерминированный **рецепт** (фиксированная последовательность по типу задачи):
`detect_object_type → (на каждый элемент) classify → create_work_breakdown →
calculate_concrete_works → export_soupis`.

- **Контекст объекта.** Тип определяется один раз (`detect_object_type`), кладётся в кэш
  по коду SO и **прокидывается** (`object_code` + `object_type`) в classify/breakdown.
  Это закрывает дыру: W3/W3b сделали это возможным, но живой путь контекст не передавал.
- **Нюанс-хук.** Когда шаг возвращает противоречие / низкую confidence / OVĚŘIT —
  **один** LLM-вызов-решение через существующий разумник (**Vertex Gemini**): на вход —
  приоритет из `so_merger` + сам конфликт; на выход — структурное решение
  `{action, chosen_source, chosen_value, reason}`. **LLM не считает числа.**
  `action=stop_gate` → поднять HITL через `stage_gating`.
- **LLM-провайдер — Vertex Gemini.** Уже живой, без AWS-кредов и без смены региона.
  **НЕ Bedrock.** Claude через Vertex Model Garden — опциональный апгрейд позже
  (нужен регион `europe-west1`/`us-east5` + включение Model Garden); сейчас не нужен,
  задача-решение мелкая, Gemini справляется.
- **OTel-трейс.** Один root-span запроса → child-span на каждый tool-call (приём из
  ADK-спайка, ~20 строк).
- **Replay.** LLM-решение **записывается в аудит/сессию** и при повторном прогоне
  **переиспользуется** (не вызывается заново) — детерминированный костяк сохраняется.
