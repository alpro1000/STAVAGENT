# STAVAGENT — Мантра для каждой сессии Claude Code

> Вставить в начало КАЖДОГО промпта перед описанием задачи.

---

## Перед тем как написать хоть одну строку кода:

### 1. Прочитай репо
```
Прочитай структуру всего репозитория alpro1000/concrete-agent.
Найди все файлы которые относятся к текущей задаче.
Не начинай писать пока не понимаешь что уже существует.
```

### 2. Определи naming
```
Все имена (переменные, функции, классы, файлы, таблицы, 
endpoint'ы) — ТОЛЬКО из существующих конвенций в репо.
Если в репо snake_case — пиши snake_case.
Если в репо файлы называются xlsx_komplet — не называй свой xlsx-komplet-parser.
Если таблица называется project_documents — не создавай portal_documents.
```

### 3. Не дублируй
```
Перед созданием ЛЮБОГО нового:
- парсера → проверь существующие парсеры
- утилиты → проверь utils/helpers
- модели/схемы → проверь models/schemas
- endpoint'а → проверь существующие роуты
- таблицы → проверь текущую схему БД

Если нашёл существующий код который делает 70% того что нужно — 
расширь его. Не пиши параллельный.
```

### 4. Встраивайся
```
Новый код должен выглядеть так, будто его написал тот же 
разработчик что писал весь остальной репо.
Не создавай новую архитектуру рядом со старой.
Core↔Kiosk паттерн — не менять.
```

## Конкретно проверить в репо перед началом работы:

| Что проверить | Зачем |
|---|---|
| `xlsx_komplet` parser | Как парсит Excel, какие поля, какой формат output |
| `xlsx_rtsrozp` parser | То же для второго формата |
| `URS_MATCHER_SERVICE` | API endpoint, формат request/response, где вызывается |
| `otskp.db` | Структура SQLite, как делается lookup, какие поля |
| `detect_document_type()` | Regex маркеры для filename и content |
| `NormIngestionPipeline` | Слои (regex→Gemini→Perplexity), порядок, confidence |
| `RegexNormExtractor` | Существующие regex паттерны — переиспользовать |
| `DocumentSummary` | Pydantic схема, какие поля, как заполняется |
| `project.json` | Текущая структура — куда добавлять новые данные |
| Таблицы PostgreSQL | Не создавать дубликаты существующих таблиц |
| `app/api/` роуты | Конвенции endpoint'ов, middleware, auth |
| `confidence scoring` | Regex=1.0, OTSKP db=1.0, URS matcher=0.80, AI=0.70 |

## Принципы STAVAGENT (не нарушать):

1. **Детерминизм прежде AI** — если можно regex, не используй LLM
2. **Core отделён от UI** — Core API не знает про фронтенд
3. **Confidence scoring обязателен** — каждый вывод имеет confidence
4. **Данные с высоким confidence не перезаписываются данными с низким**
5. **Multi-provider AI** — Gemini Flash (быстро) → Claude Sonnet (точно) → Perplexity (нормы)
6. **Render НЕ ИСПОЛЬЗУЕТСЯ** — всё на Cloud Run + Vercel

## Формат работы:

```
1. AskUserQuestion — задай 5-7 вопросов ПЕРЕД написанием кода
2. Покажи план с gates — не делай следующий этап без подтверждения
3. Пиши тесты — unit тесты с моками, без реальных HTTP/DB
4. После каждого этапа — краткий отчёт что сделано и что дальше
```
